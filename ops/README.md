# Эксплуатация: бэкапы и восстановление

## Что попадает в бэкап

Один `pg_dump` покрывает **всё**: медзаписи, учётки и **файлы документов** —
блобы лежат в таблице `blob` (BYTEA), отдельного файлового хранилища нет.

Два свойства дампа, важных для медданных:

- **Медицинская часть в дампе зашифрована** (DEK под KEK). Без `KEK` из `.env`
  восстановить её невозможно — дамп сам по себе бесполезен.
- **Идентификационная часть — открытым текстом** (ФИО, email). То есть утечка
  дампа = утечка ПДн, даже без KEK.

Отсюда правило: **`KEK` хранить отдельно от бэкапов** (менеджер паролей), а сами
дампы — как персональные данные.

## Расписание

Ежедневный дамп ставится в cron на VPS:

```bash
crontab -e
```
```cron
15 3 * * * bash /opt/apps/shop/ops/backup.sh >> /opt/apps/backups/backup.log 2>&1
```

Плюс дамп делается автоматически **перед каждым выкатом** (CD вызывает тот же
скрипт с префиксом `pre-<тег>`) — чтобы кривая миграция откатывалась.

Хранится по 14 последних дампов каждого вида (`KEEP` меняет глубину).

## Проверить, что расписание живо

Молча сломавшийся бэкап — классика жанра, поэтому скрипт пишет маркер:

```bash
cat /opt/apps/backups/LAST_SUCCESS     # дата последнего успешного дампа
ls -lh /opt/apps/backups/              # размеры: резкое падение = повод смотреть
tail -20 /opt/apps/backups/backup.log
```

Если `LAST_SUCCESS` старше суток — расписание не работает, разбираться.

## Восстановление

Дамп снимается с работающей БД (`pg_dump` даёт консистентный снимок), а вот
заливать его нужно в **чистую** базу, иначе конфликты с существующими данными.

```bash
cd /opt/apps/shop
docker compose stop api worker           # чтобы никто не писал во время заливки

# пересоздать базу начисто
docker compose exec -T postgres psql -U shop -d postgres \
  -c "DROP DATABASE shop WITH (FORCE)" -c "CREATE DATABASE shop"

# залить дамп
gunzip -c /opt/apps/backups/daily-ГГГГММДД-ЧЧММСС.sql.gz \
  | docker compose exec -T postgres psql -U shop shop

docker compose up -d api worker
```

Проверить после восстановления: открывается ли карта, виден ли документ
(это докажет, что и блобы, и ключи на месте).

⚠️ Восстановление на **новом** сервере требует не только дампа, но и того же
`KEK` в `.env` — иначе медданные останутся нечитаемыми.

## Чего эти бэкапы НЕ закрывают

Дампы лежат на том же VPS, что и БД. Это защищает от кривой миграции,
случайного удаления и бага приложения, но **не** от потери самого сервера
(отказ диска, проблемы с провайдером, компрометация). Для этого нужна копия
вне сервера — см. «Дальше» ниже.

## Дальше (не сделано)

- **Копия off-site** (S3/B2 через rclone или другой сервер). Дамп содержит ПДн,
  поэтому наружу — только зашифрованным: например `age -r <публичный ключ>`,
  тогда сервер умеет только писать бэкапы, но не читать старые.
- **Тестовое восстановление** по расписанию — раз в квартал поднять дамп в
  отдельную БД и убедиться, что он живой.
